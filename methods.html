Methods · Code · Reproducibility

How this analysis works

Every map, chart, and number on the main page comes from one of three integrated pipelines: a Google Earth Engine classification stack, a config-driven arcpy ArcGIS Pro pipeline, and a curated set of public health datasets. This page walks through each, links to all source code, and surfaces the methodological caveats — including a serious classifier-failure problem on the West Coast that any reader of the main story should know about.

Critical caveat: the classifier fails in St. James and St. John

The v8c Landsat classifier achieves only 18.6% accuracy and 0.03 kappa in St. James — the Platinum Coast itself. St. John performs even worse (6.5% / 0.03). This was discovered by holding out each parish in turn as a test set during cross-validation. The full table is below, and the underlying CSV is SpatialCV_byParish_v8c.csv.

Any per-parish leisuring, urbanisation, or transition rate from the v8c rasters for St. James and St. John should be treated as a lower-bound, indicative estimate rather than a measurement. The 10 m Sentinel-2 classification (Classification_10m_v8f) uses a different sensor and training set; cross-checking against it is the standard mitigation.

The three pipelines

1. Google Earth Engine classification — code/gee/

Five JavaScript files that build Landsat composites and train Random Forest classifiers. Paste each into code.earthengine.google.com and click Run.

01_composite_1984_1986.jsBaseline composite, 1984–1986 (Landsat 5)
02_composite_2020.jsCurrent composite, 2020 (Landsat 7/8)
03_classify_and_change.jsBuilds both composites, trains Random Forest on 2020 + Copernicus reference, classifies both periods, exports change map
04_classify_with_challenge_improvements.jsSame as 03, with 500 trees and 300 samples per class (challenge.html recommendations)
05_classify_SR_5class.jsSurface-reflectance 5-class variant
LandTrendr_Walkthrough.mdGUI walkthrough for the LandTrendr time-series module on Barbados

The methodology follows the SERVIR-Amazonia land-cover mapping tutorial verbatim through Steps 1–2, then extends it with a binary change map and a From-To transition raster.

2. ArcGIS Pro pipeline (config-driven arcpy) — code/pipeline/

Twelve phases that automate the entire ArcGIS Pro protocol. Swap a single YAML file (config/barbados.yaml ↔ config/newfoundland.yaml) to re-run the whole thing on a different study area.

PhaseWhat it does
0Inventory — checks every input file the config references exists
1Project setup — folders, .gdb, feature dataset
2Ingest GEE script-04 outputs — project rasters, extract NDVI / MNDWI / temp bands
2bIngest the v8 series from Google Drive — 5 epoch classifications, CCDC, BecameUrban, SugarcaneConversion, ManicuringIndex, NDVI trend, S1 SAR stack, parish CV table
3Ingest Esri/IO Sentinel-2 annual LC — build a multidimensional mosaic dataset
4Ingest vectors — parishes, roads, coastline, buildings, listings (CSV with lat/lon)
5Harmonise — reclassify Esri/IO to Copernicus 8-class, compute 2020 agreement
6Core analyses — transition matrix, leisuring rate, coastline change, urban expansion, recent change
6bv8 analyses — 4 consecutive-epoch transitions, BecameUrban / SugarcaneConversion / CCDC break-year / ManicuringIndex / NDVI trend, all by parish; uncertainty mask
7Listings integration — extract every raster covariate (incl. v8) onto each 7thHeaven listing; spatial-join admin
8Spatial statistics — GLR, Exploratory Regression, GWR, Optimized Hot Spots
9Layouts — pre-load every derived layer into a new map in the .aprx; export layout PNGs

From ArcGIS Pro's Python Command Prompt:

cd code\pipeline
pip install -r requirements.txt
python run_pipeline.py --config config/barbados.yaml

# or any subset
python run_pipeline.py --config config/barbados.yaml --phase 2b
python run_pipeline.py --config config/barbados.yaml --phases 2-6b

To re-run on Newfoundland: fill in code/pipeline/config/newfoundland.yaml and swap the --config flag. Zero code changes — class palette, target CRS, paths, Esri-to-Copernicus remap, regression variables, all live in YAML.

3. Health data — data/health/ + docs/

Curated extraction from three sources, scoped to the project window:

barbados_cvd_timeseries_2013_2022.csv10-year national CVD time series — AMI & stroke, cases / incidence / mortality, men & women, ASIR + ASMR. From the Barbados National Registry CVD Annual Report 2024.
barbados_ncd_risk_factor_prevalence.csvLong-format table of HoTNS / GSHS / GYTS / BNR risk-factor prevalence values.
HEALTH_DATA_SYNTHESIS.mdSynthesis of all three health sources mapped to the land-use × tourism × health framing.
census2010_ED_health_columns_guide.mdColumn-by-column guide to the 2010 census ED demographics CSV (disease counts, dwelling-vintage cohorts, tourism-pressure proxies, SES confounders).

Design principles

  • Idempotent. Every phase checks if its outputs already exist and skips them. Safe to rerun freely.
  • Config-driven. Every region-specific value (CRS, paths, class palettes, regression variables) lives in YAML. The code never assumes "Barbados".
  • Failure-tolerant. Each analysis function is wrapped in try/except so one missing input doesn't abort the rest of the phase.
  • Logged. Every run writes a timestamped log to code/pipeline/logs/; the parish CV warnings are reproduced in every Phase 2b log.
  • Reproducible end-to-end. The GEE scripts are reader-pasteable, the pipeline is one-command, the data files are static CSV / GeoTIFF, and the site is a static GitHub Pages deploy.

On causation

This project demonstrates an association and a mechanism, not a proven cause. The land-use → obesogenic-environment → cardiovascular-disease pathway is consistent with the WHO Global Syndemic framing (Swinburn et al., Lancet 2019) but confirming it for Barbados specifically requires sub-national CVD incidence and time-varying controls for wealth, age structure, and healthcare access. The pipeline is designed to accept those data when (and if) they become available — the health-data synthesis documents exactly what's missing and where to get it.
